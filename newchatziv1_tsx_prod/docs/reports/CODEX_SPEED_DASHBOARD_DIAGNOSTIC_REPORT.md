## Findings

- Fast Responder was present and connected, but it did not honor `AI_FAST_RESPONDER_ENABLED`. It now does.
- Fast Responder is used before Knowledge/Mastra in both `src/lib/ai.ts` and `src/mastra/workflows/ai-reply.workflow.ts`.
- Fast Responder replies are generated by the configured AI provider and are not hardcoded greeting/thanks/out-of-scope replies.
- Mastra tools were present and attached to `customerSupportAgent`, but runtime generation was failing with the registered placeholder model. The workflow now creates a runtime customer-support agent with the DB-resolved model.
- Mastra CRM tools are connected: search knowledge, get customer profile, create/update ticket, create/update lead, summarize conversation.
- `summarizeConversation` now clamps `limit` safely.
- AI worker concurrency is env-driven and aligned to default `4`.
- Egress worker concurrency is env-driven and aligned to default `8`.
- Outbound worker concurrency is env-driven and aligned to default `8`.
- Core routing worker concurrency is env-driven and aligned to default `6`.
- PM2 config includes AI fast responder flags and worker concurrency defaults.
- `sidebar-counts` is mounted inside `sidebar.tsx`.
- Sidebar counts API returns the requested shape and now uses active conversation filters with tenant-safe ObjectId casting.
- Tickets create/update leads through `syncLeadFromTicket`; leads now carry `ticketId`, `intent`, `status`, and `lastMessageAt`.
- Dashboard tickets pagination is real; API and page default to limit `10`.
- Realtime assistant messages are published after saving and now start with `deliveryStatus: "queued"`.
- Egress/outbound updates delivery status to `sending` then `sent`.
